# Evo5
Network Management Server

## Purpose
The goal is to build a headless network management server that can act as a bridge between the cruel world outside and a safe haven subnet you want to work within.

## History
My first machine performing this role was a [Compaq Evo D510](https://en.wikipedia.org/wiki/Compaq_Evo). This managed the subnet in my shed and had a wireless uplink to the house. The network that it oversaw contained two 6' 19" racks with various Unix boxes from yesteryear. I replaced it with a [Raspberry Pi 3](https://www.raspberrypi.com/products/raspberry-pi-3-model-b/) along with a [TP-Link Wireless Nano Router](https://www.tp-link.com/au/home-networking/wifi-router/tl-wr802n/). This is now being replaced by a [Raspberry Pi 5](https://www.raspberrypi.com/products/raspberry-pi-5/) along with a [Raspberry Pi USB WiFi Dongle](https://www.raspberrypi.com/products/raspberry-pi-usb-wifi-dongle/).

## Network Architecture
There are three subnets.

* [LAN](https://en.wikipedia.org/wiki/Local_area_network) - Main subnet using ethernet connected to a simple 5 port switch
* [WAN](https://en.wikipedia.org/wiki/Wide_area_network) - Uplink subnet is a WiFi client of the accomodation or office
* [WLAN](https://en.wikipedia.org/wiki/Wireless_LAN) - Local subnet provided by the Wifi dongle for local devices

The LAN subnet and WLAN subnet will route through the WLAN using NAT/MASQUERADE. Local devices can either use WiFi or ethernet and be able to access other local devices.

## Implementation
The [Raspberry Pi OS Lite Trixie](https://www.raspberrypi.com/software/operating-systems/) is used as the host OS. Where possible existing management configuration and tooling are used rather than custom scripting.

### Preparation
This following componentns are needed
* Raspberry Pi 5 + PSU
* SD Card, 16GB minimum recommended
* Ethernet switch and cables for LAN
* Supported Wifi USB Dongle for WLAN
* Laptop or other to write SD card and act as SSH client
* A Wifi hotspot to share between laptop and Raspberry Pi during bootstrap

### Bootstrapping
The [Raspberry Pi Imager](https://www.raspberrypi.com/news/raspberry-pi-imager-imaging-utility/) is used to write the SD card. As the device is headless the following configuration is required at this stage.
* Raspberry Pi OS Lite - we have no need for a GUI
* Configure networking so it works out of the box - suggest Wifi to a Hotspot on your phone
* Configure SSH acceess without passwords, use SSH keys for authentication
* Configure your username, password and hostname

Once the device boots from the SD card it should connect to the configured network and be available for login.

### Login in for the first time
With a laptop connected to the same network you should be able to login using `ssh` with _hostname_.local and the configured key. Look in `/etc/netplan` for the configuration generated by the bootstrap process.

### Configure the WAN netplan yaml
As we have two Wifi devices we need to identify wlan0 by using the MAC address.

```
# ifconfig wlan0 | grep ether
        ether 88:a2:9e:ca:16:5a  txqueuelen 1000  (Ethernet)
```

Add the macaddres

```
#  cat /etc/netplan/90-NM-1a8bdcba-dd00-4a7d-9365-683ec3007b35.yaml
network:
  version: 2
  wifis:
    wlan0:
      renderer: NetworkManager
      match:
        macaddress: "88:A2:9E:CA:16:5A"
      dhcp4: true
      access-points:
        "The Black Gate":
          auth:
            key-management: "psk"
            password: "One does not simply walk into Mordor"
          networkmanager:
            uuid: "1a8bdcba-dd00-4a7d-9365-683ec3007b35"
            name: "netplan-wlan0-mordor"
            passthrough:
              proxy._: ""
      networkmanager:
        uuid: "1a8bdcba-dd00-4a7d-9365-683ec3007b35"
        name: "netplan-wlan0-mordor"
```

Reboot and confirm that the Wifi uplink is using wlan0.

### Configure the LAN netplan yaml
We want this machine to manage the local ethernet with DHCP. In order to do this it first needs a subnet with a static IP.

```
#  cat /etc/netplan/90-NM-75a1216a-9d1a-30cd-8aca-ace5526ec021.yaml
network:
  version: 2
  ethernets:
    eth0:
      renderer: NetworkManager
      match: {}
      addresses:
      - "10.1.2.1/24"
      networkmanager:
        uuid: "75a1216a-9d1a-30cd-8aca-ace5526ec021"
        name: "netplan-eth0"
        passthrough:
          proxy._: ""
```

### Configure the WLAN hotspot using nmcli
The local Wifi needs to be created directly with `nmcli` rather than the netplan yaml. This has the advantage that hostapd is not required.

First, determine the encoded password for your new local hotspot.

```
$ wpa_passphrase MYSSID MYPASSPHRASE
network={
	ssid="MYSSID"
	#psk="MYPASSPHRASE"
	psk=ENCRYPTED_PSK_IS_HERE
}
```

The encrypted PSK can then be configured directly. This also defines the IP address and subnet for the hotspot.

```
$ nmcli connection add \
    con-name ap0 \
    type wifi \
    ifname wlan1 \
    ssid MYSSID \
    802-11-wireless.mode ap \
    802-11-wireless.band bg \
    wifi-sec.pairwise ccmp \
    wifi-sec.key-mgmt wpa-psk \
    wifi-sec.psk ENCRYPTED_PSK_IS_HERE \
    ipv4.method static \
    ipv4.address 10.1.3.1/24 \
    802-11-wireless-security.pmf 1 \
    autoconnect yes

$ nmcli connection up ap0
```

There were a few problems arriving at this solution. You need to confirm that the hotspot does come up and does not timeout with wpa_supplicant.

|Option|Comment|
|------|-------|
|802-11-wireless-security.pmf 1|was added to prevent timeout with wpa_supplicant|
|wifi-sec.proto rsn|was not used as it caused password failures|

Verify that the new SSID is visible from a handheld device.

### Configure DHCP with dnsmasq
DHCP is configured using a custom `dnsmasq` configuration file.

```
# cat /etc/dnsmasq.d/dnsmasq.evo5.conf
interface=eth0
interface=wlan1
dhcp-range=10.1.2.128,10.1.2.250,255.255.255.0,12h
dhcp-range=10.1.3.128,10.1.3.250,255.255.255.0,12h
dhcp-option=option:netmask,255.255.255.0
bind-interfaces
```

This defines both the LAN and WLAN subnets.

### Configure NAT and routing with iptables
The following commands can be put in `/etc/rc.local`

```
iptables -t nat -A POSTROUTING -o wlan0 -j MASQUERADE
iptables -A FORWARD -i eth0 -j ACCEPT
iptables -A FORWARD -i wlan1 -j ACCEPT
```

These rules do the following

* Use NAT when routing to the WAN
* Route from LAN to WAN
* Route from WLAN to WAN

Reboot and confirm all comes back and the `route` and `nmcli` connections look consistent. Refer to the tables shown in the [End State](#end-state) section.

## Final wrap-up

Follow the following tests

### Confirm correct WAN is configured
If you were using a temporary mobile phone hotspot during bootstrapping, reconfigure the WAN netplan yaml as required.

### Confirm that LAN works
Using laptop and ethernet confirm that LAN has access to device and the internet

### Confirm that WLAN works with laptop
Connect laptop via Wifi to local WLAN hotspot and confirm access to both device and the internet

### Confrm that WLAN works with handheld devices
Connect mobile telephone to WLAN hotspor, confirm able to login wih password and access internet. Use 

```
# nmcli device wifi show
```

to display QR code to test phone access.

## Emergency Recovery

With the Raspberry Pi Imager it is simple to create multiple boot images. If you make a mistake in configuration you can recover by
* using the power switch to safely shut down
* boot from a second boot device
* mount the offending media
* fix the problem
* shutdown
* revert to the original boot media

```
# lsblk
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sda      8:0    1 14.3G  0 disk
├─sda1   8:1    1  512M  0 part /boot/firmware
└─sda2   8:2    1 13.8G  0 part /
sdb      8:16   1 59.4G  0 disk
├─sdb1   8:17   1  512M  0 part
└─sdb2   8:18   1 58.9G  0 part
zram0  254:0    0    0B  0 disk
# mount /dev/sdb2 /mnt
# cd /mnt/etc/
#
```
